# Gemini 사용 규칙

## 1. 명확하고 구체적인 요청
- **나쁜 예:** "버그 수정해줘."
- **좋은 예:** "`src/main/kotlin/Main.kt` 파일의 `calculate` 함수에서 `NullPointerException`이 발생하는 문제를 해결해줘."

## 2. 충분한 맥락(Context) 제공
- 코드 일부만 보여주기보다는 관련된 파일 구조, 사용 중인 라이브러리, 프로젝트의 전체적인 목표 등을 함께 알려주시면 더 정확한 답변을 드릴 수 있습니다.
- "현재 프로젝트는 Gradle 기반의 Kotlin 프로젝트야." 와 같이 프로젝트 환경을 알려주는 것이 좋습니다.

## 3. 점진적으로 결과물 완성하기
- 첫 결과물이 완벽하지 않을 수 있습니다. 구체적인 피드백을 통해 원하는 결과물로 함께 만들어가는 것이 좋습니다.
- "이 부분은 좋은데, 저 부분은 `if`문 대신 `when`을 사용해서 수정해줘." 와 같이 구체적으로 지시해주세요.

## 4. 도구 적극적으로 활용하기
- 저는 파일을 읽고, 쓰고, 검색하고, 셸 명령을 실행하는 등 다양한 도구를 사용할 수 있습니다.
- "현재 폴더의 모든 `.kt` 파일에서 'import java.util'을 사용하는 곳을 찾아줘." 와 같이 도구 사용을 직접 지시하면 좋습니다.

## 5. 안전 확인
- 제가 파일을 수정하거나 셸 명령을 실행하기 전에는 항상 사용자에게 확인을 요청합니다. 어떤 동작이 수행되는지 꼭 확인하고 승인해주세요.

## 6. 변경 사항은 항상 검토
- 제가 생성하거나 수정한 코드는 **반드시** 직접 검토하고 테스트해야 합니다. 저는 개발자를 돕는 조수이며, 최종 책임은 개발자에게 있습니다.

## 7. 프로젝트 규칙(Convention) 명시
- 프로젝트에 특별한 코딩 스타일이나 컨벤션이 있다면 미리 알려주세요. 일관성 있는 코드를 생성하는 데 큰 도움이 됩니다.
